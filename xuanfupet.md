# Android 桌面悬浮宠物原型 \- 优化后完整实现方案

## 文档信息

- 项目：宠物时间锁 Flutter App

- 目标：Android 全局悬浮宠物，提升用户陪伴情绪价值

- 基准方案：原 \[xuanfu\.md\]\(xuanfu\.md\) 实现计划

- 本文档：全量修复风险、补充体验 / 技术优化、明确待决策项

## 1\. 背景与目标

### 1\.1 业务背景

为提升产品传播性与用户情绪陪伴时长，新增桌面全局悬浮宠物，宠物可脱离 App 独立常驻桌面；专注完成、手机使用超时、宠物进化时自动弹出互动提醒。

- 技术限制：Android 支持全局悬浮窗，iOS 无系统级悬浮能力，仅能使用小组件 / 活动通知降级替代

- 技术底座：现有 Flutter 项目，基于`flutter_overlay_window`快速搭建原型

### 1\.2 核心优化目标

1. 修复原方案**状态同步丢失、多 Android 版本权限兼容、进程被杀不同步**等技术风险

2. 补充防误触、个性化配置、权限引导、边缘吸附等用户体验短板

3. 完善性能分级策略，解决双 Flutter 引擎内存过高问题

4. 规范原生与 Dart 通信常量，提升工程可维护性

5. 补充完整测试用例、异常兜底逻辑

6. 明确所有需要产品 / 技术负责人决策的关键节点

## 2\. 技术选型与优化说明

### 2\.1 核心依赖不变

仍使用 `flutter_overlay_window: ^0.5.0` 作为悬浮窗底层引擎
优势：封装 WindowManager、独立 FlutterEngine，支持拖拽、显隐、跨进程通信，大幅减少原生手写 Overlay Service 工作量，适合快速原型验证

### 2\.2 新增优化机制

1. **PetState 版本号同步机制**：解决跨引擎状态不一致、消息丢失问题

2. **分级渲染模式**：高性能动态宠物 / 低内存原生静态图标自动切换

3. **DeepLink \+ 本地缓存 PendingAction**：App 被杀死后精准恢复对应操作页面

4. **Android 全版本权限分级适配逻辑**

5. **数据库读写互斥锁**，规避并发脏读

6. **拖拽区域安全裁剪**：适配刘海屏 / 挖孔屏，自动边缘吸附

## 3\. 整体架构与状态同步策略（优化版）

### 3\.1 数据源规则（单一可信源不变，增加一致性保障）

1. 唯一数据源：本地 SQLite `DatabaseHelper`

2. PetState 新增 `version` 自增字段，宠物任意属性变更时版本 \+ 1

3. 主 App 更新宠物状态流程：

    - 更新 DB 数据 \+ version 自增

    - 调用 `FlutterOverlayWindow.shareData` 发送 `refresh_pet` 消息

    - 5s 未收到 Overlay 回执则自动重发一次刷新指令

4. Overlay 读取逻辑：

    - 启动 / 收到刷新指令时读取本地 DB version

    - 若本地缓存 version 与 DB 不一致，完整重载宠物状态；否则跳过读取，减少 IO 开销

5. 数据库并发保护：所有宠物状态读写加单例互斥锁，使用 SQLite 事务隔离读写

### 3\.2 跨进程通信协议（扩展优化）

#### 3\.2\.1 主 App → Overlay

|动作|通信方式|携带参数 Payload|优化补充|
|---|---|---|---|
|显示 / 隐藏悬浮窗|showOverlay / closeOverlay|position, size, renderMode|新增 renderMode 区分动态 / 静态渲染|
|常规刷新宠物状态|shareData|`{"action":"refresh_pet","version":xxx}`|携带版本号，减少无效刷新|
|触发弹窗（专注 / 超时 / 进化）|shareData|`{"action":"show_trigger","trigger":"xxx","message":"文案","duration":8000}`|时长支持动态配置|

#### 3\.2\.2 Overlay → 主 App

|动作|通信方式|携带参数|优化补充|
|---|---|---|---|
|点击菜单拉起 App 执行操作|MethodChannel \+ DeepLink|`payload:feed/play/focus...`|1\. 写入 SharedPreferences 暂存待执行操作<br>2\. DeepLink 携带路由参数，冷启动直接跳转对应页面|
|拖拽位置持久化|SharedPreferences|overlay\_x, overlay\_y|拖拽结束自动裁剪坐标至屏幕安全区|

### 3\.3 Overlay 与主 App 操作流转

1. Overlay 点击操作 → MethodChannel 发送指令

2. 原生层拼接 DeepLink 拉起应用

3. 应用冷 / 热启动后，`main.dart` 读取缓存 PendingAction 并自动路由至对应页面

4. 主 App 完成喂食 / 专注等操作后更新 DB，推送刷新指令同步悬浮宠物

## 5\. UI/UX 交互规范（优化完整版）

### 5\.2 展开菜单态

1. 宠物上方弹出圆角白色操作菜单，5s 无操作自动收起，点击空白遮罩立即收起

3. 可配置项：设置页可单独关闭「专注完成 / 超时 / 进化」三类提醒，自定义弹窗显示时长（5/8/10 秒）

4. 交互：8s 自动恢复折叠态；点击直接展开操作菜单

### 6\.1 Android 版本权限分级

|Android 版本|悬浮窗权限逻辑|前台服务要求|补充权限|

### 6\.2 权限申请完整流程

1. 用户点击开启悬浮宠物 → 弹出自定义引导弹窗，说明「悬浮窗用于桌面宠物陪伴、专注提醒」\+ 简易操作示意图

3. 用户拒绝权限：

    - 不崩溃，保留设置页悬浮宠物开关入口

    - 提示文案：权限未授予，仅可在 App 内与宠物互动，无法桌面悬浮

    - 提供一键重新授权按钮

### 6\.3 iOS 降级方案优化

原方案仅提示不支持悬浮窗，优化后：

1. 设置面板悬浮宠物开关置灰，展示引导文案：iOS 系统限制无法全局悬浮，可添加桌面小组件获取宠物状态陪伴

2. 附带小组件添加步骤图文指引，点击跳转系统小组件添加页面

3. 小组件规划（后续迭代）：展示宠物当前状态、快捷进入专注 / 喂食页面

## 7\. 性能分级策略（解决双引擎内存风险）

### 7\.1 监控指标

通过`overlay_performance_service`持续采集 Overlay 引擎：

- 内存占用、CPU 持续使用率

### 7\.2 分级渲染规则

1. **高性能模式（默认）**：Flutter 完整动态动画宠物，适合中高端机型

2. **低性能模式（自动降级）**

    - 触发阈值：Overlay 内存持续 \> 100MB / CPU 占用 \> 20% 维持 10s

    - 表现：切换为 Android 原生静态宠物图标，保留拖拽、点击拉起功能，移除全部动画，大幅降低双引擎内存占用

3. 原型阶段仅做监控埋点，上线前根据真机数据开启自动降级

## 8\. 风险点与优化兜底方案

|风险点|原方案缺陷|优化后缓解措施|
|---|---|---|
|Android14 \+ 系统后台强杀悬浮进程|仅声明前台服务|1\. FOREGROUND\_SERVICE\_SPECIAL\_USE 前台服务常驻通知<br>2\. 进程被回收后，用户再次打开 App 自动恢复悬浮窗位置|
|悬浮权限被用户永久拒绝|仅简单引导|自定义引导弹窗 \+ 重新授权按钮，不强制功能，保留 App 内宠物体验|
|iOS 无全局悬浮能力|仅文字提示|完善小组件引导文案，提供替代陪伴方案|
|双 Flutter 引擎内存过高|仅后续备选方案，无监控|新增性能监控 \+ 自动静态渲染降级|
|跨进程消息丢失，宠物状态不同步|无消息回执、无版本校验|1\. PetState 版本号对比刷新<br>2\. 刷新指令 5s 无回执自动重发<br>3\. Overlay 启动强制全量读取一次 DB|
|SQLite 并发读写脏读|无锁机制|数据库读写加单例互斥锁，使用事务更新宠物状态|
|拖拽遮挡状态栏 / 刘海屏|仅简单边界限制|原生获取 DisplayCutout，拖拽终点自动避让异形屏幕区域，自动吸附屏幕边缘|
|App 被系统杀死，点击悬浮菜单无法跳转对应页面|仅 MethodChannel 拉起|DeepLink \+ 本地缓存 PendingAction，冷启动自动路由至目标页面|
|用户频繁弹窗提醒产生打扰|无开关控制|设置面板分项开关三类触发弹窗，自定义弹窗时长、关闭悬浮提醒总开关|
